# Обзор безопасности (Security Review)

## 1. Область и метод

Обзор охватывает веб-приложение и API RegionLens: аутентификацию и доступ, транспорт и
заголовки, защиту от типовых атак, ограничение частоты, обращение с секретами, приватность
данных и безопасность зависимостей. Метод — анализ конфигурации и кода, подкреплённый
автоматическими тестами (`tests/test_security_settings.py`, `tests/test_roles.py`,
`tests/test_api_token.py`, `tests/test_auth.py`) и статическим аудитом зависимостей
(`pip-audit`).

Модель угроз ограничена: исходные данные — открытая статистика Росстата, персональные данные
минимальны (учётные записи пользователей). Основные защищаемые активы — целостность аналитики,
корректное разграничение доступа к кабинету и админ-панели, недопущение злоупотреблений API.

## 2. Аутентификация и управление доступом

- **Роли.** Трёхуровневая модель `viewer ⊂ analyst ⊂ admin` на группах Django; проверки через
  разрешения и декораторы. Аналитические эндпойнты открытых данных публичны намеренно (открытые
  данные), приватны — кабинет (`@login_required`) и админ-панель. Контракт зафиксирован тестами
  ролей и доступа.
- **Сессии.** Стандартная сессионная аутентификация Django; пароли хранятся в виде хешей
  штатными хешерами (PBKDF2).
- **Токены API.** Личный токен для программного доступа хранится **только в виде SHA-256-хеша**;
  сырой ключ показывается один раз при выдаче и не восстановим. Проверка — хешированием
  предъявленного ключа и сравнением с БД. Отзыв очищает хеш. Покрыто тестами.
- **Разделение методов.** Для браузера — сессия (с CSRF), для API-клиентов — токен без cookies.

## 3. Транспорт и защитные заголовки

В боевом режиме (`DEBUG=False`) включаются строгие флаги (проверяется
`test_prod_enables_secure_flags`):

| Механизм | Значение |
|---|---|
| Принудительный HTTPS | `SECURE_SSL_REDIRECT=True` + доверие `X-Forwarded-Proto` от прокси |
| HSTS | 1 год, `includeSubDomains`, `preload` |
| Secure-cookies | `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE` = True |
| HttpOnly / SameSite | `SESSION_COOKIE_HTTPONLY=True`, `SameSite=Lax` |
| Анти-MIME-sniffing | `SECURE_CONTENT_TYPE_NOSNIFF=True` (в dev и prod) |
| Анти-кликджекинг | `X_FRAME_OPTIONS="DENY"` (в dev и prod) |

В режиме разработки строгие транспортные флаги ослаблены, чтобы `runserver` работал по HTTP
(`test_dev_keeps_flags_relaxed`), но анти-sniffing/анти-кликджекинг и HttpOnly активны всегда.

## 4. Защита от типовых атак

- **CSRF.** `CsrfViewMiddleware` включён; формы кабинета передают CSRF-токен. Read-only API
  использует безопасные методы (GET); токен-аутентификация не полагается на cookies, поэтому
  не подвержена CSRF.
- **SQL-инъекции.** Запросы к витрине DuckDB **параметризованы** (плейсхолдеры `?`, значения
  передаются отдельным списком). Пользовательский ввод (год, ОКАТО, схема, строка поиска)
  никогда не конкатенируется в SQL. Немногие f-строки в запросах подставляют лишь **списки
  колонок из внутренних констант**, а не пользовательские данные. Дополнительно соединение с
  DuckDB — **read-only**, что исключает модификацию данных даже гипотетически.
- **XSS.** Серверный рендеринг на шаблонах Django с автоэкранированием по умолчанию; «сырой»
  вывод пользовательского ввода не практикуется.
- **Кликджекинг.** `X-Frame-Options: DENY` + `XFrameOptionsMiddleware`.

## 5. Ограничение частоты (anti-abuse)

Троттлинг DRF: аноним — 120/мин на IP, по токену — 600/мин (значения конфигурируются
окружением). Защищает от исчерпания ресурсов агрессивным источником; при превышении —
`429 Too Many Requests`. Действенность подтверждена нагрузочным замером (см.
`docs/testing/load_test_report.md`, раздел 5).

## 6. Секреты и конфигурация

- `SECRET_KEY`, учётные данные БД, хосты и флаги задаются через переменные окружения; в
  репозитории секретов нет.
- **Предохранитель:** запуск в боевом режиме с дефолтным небезопасным `SECRET_KEY`
  останавливается исключением `ImproperlyConfigured` (`test_prod_requires_real_secret_key`).
- Пути к витрине DuckDB и каталогу моделей также конфигурируются окружением (без хардкода).

## 7. Данные и приватность

- Обрабатываемые аналитические данные — открытые (Росстат). Персональные данные ограничены
  учётными записями (логин, e-mail, хеш пароля).
- Действия в кабинете (экспорт, выдача/отзыв токена и т.п.) фиксируются в журнале аудита —
  прослеживаемость без хранения избыточных данных.

## 8. Безопасность зависимостей

Аудит `pip-audit` по `requirements.txt` выявил **13 известных уязвимостей в 5 транзитивных
пакетах**, подтягиваемых преимущественно `mlflow` (не прямой код приложения и не «горячие»
пути Django):

| Пакет | Версия | Исправлено в |
|---|---|---|
| `aiohttp` | 3.14.0 | 3.14.1 |
| `cryptography` | 48.0.0 | 48.0.1 |
| `starlette` | 1.2.1 | 1.3.1 |
| `pydantic-settings` | 2.14.1 | 2.14.2 |
| `diskcache` | 5.6.3 | — (следить за выпуском) |

**Рекомендации:**
1. Обновить зафиксированные версии (`uv lock --upgrade` с последующим экспортом
   `requirements.txt`) — большинство уязвимостей закрываются патч-версиями.
2. Рассмотреть переход на `mlflow-skinny` (используется только логирование моделей) —
   это сократит транзитивную поверхность (fastapi/starlette/aiohttp).
3. Включить `pip-audit` в CI как периодическую проверку.

Зарегистрировано как дефект **BR-04** (см. `docs/testing/bug_reports.md`).

## 9. Ручные и кроссбраузерные проверки

Автотесты покрывают серверную логику и заголовки. Дополнительно рекомендуется ручная проверка:
корректность защитных заголовков в ответах боевого стенда (через инструменты разработчика
браузера), поведение форм входа/выхода, отображение в актуальных версиях Chrome/Firefox/Edge.
Эти проверки выполняются на стенде и не входят в автоматический набор.

## 10. Итог

Базовая поверхность защищена в соответствии с практиками OWASP для веб-приложений: строгий
транспорт (HTTPS/HSTS), защитные заголовки, CSRF, отсутствие инъекционной поверхности
(параметризация + read-only витрина), безопасное хранение паролей и токенов (хеширование),
ограничение частоты, изоляция секретов с предохранителем. Единственная существенная
рекомендация — обновление транзитивных зависимостей (раздел 8). Критических уязвимостей в
коде приложения не выявлено.
