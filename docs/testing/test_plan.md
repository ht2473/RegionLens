# Тест-план RegionLens

## 1. Назначение

Документ описывает стратегию, уровни, область и критерии тестирования платформы RegionLens —
аналитической BI-системы социально-экономических индикаторов регионов России. Цель
тестирования — подтвердить корректность аналитических расчётов, надёжность API и веб-интерфейса,
соблюдение разграничения доступа и целевых нефункциональных характеристик.

## 2. Объект тестирования

| Подсистема | Что проверяется |
|---|---|
| Конвейер данных (ETL) | Загрузка, дедупликация, контракты схемы, качество данных |
| Аналитические расчёты | Индекс развития, доменные баллы, схемы весов, конвергенция, дисперсия, устойчивость рейтинга |
| Типология и аномалии | Кластеризация, профили типов, пространственные и структурные аномалии |
| ML-сервис | Сохранение/загрузка моделей, карточки, воспроизводимость, объяснимость (SHAP) |
| API (DRF) | Контракт ответов, версионирование `/api/v1/`, ограничение частоты, токен-аутентификация, схема OpenAPI |
| Веб-интерфейс | Серверный рендеринг страниц, личный кабинет, экспорт, локализация RU/EN |
| Доступ и безопасность | Роли (viewer/analyst/admin), аутентификация, аудит, security-настройки |
| Эксплуатация | Health-checks, сиды, соединение с DuckDB |

## 3. Уровни и стратегия тестирования

Тестирование построено по принципу «пирамиды»: широкое основание быстрых модульных тестов,
меньший слой интеграционных, вершина — нефункциональные проверки.

- **Модульные.** Чистые функции расчётов (например, `custom_ranking_from_rows`,
  `scenario_from_rows`, доменные баллы, перцентили) тестируются без БД на подготовленных
  входных данных — быстро и детерминированно.
- **Контрактные.** Схемы таблиц витрины проверяются контрактами `pandera`; тесты качества
  данных фиксируют инварианты (диапазоны лет, отсутствие дублей, коды регионов).
- **Интеграционные.** Эндпойнты API и страницы прогоняются через тестовый клиент поверх
  временной витрины DuckDB (фикстура создаёт таблицы, подменяет `DUCKDB_PATH`, сбрасывает
  кэш соединения). Проверяются статус, форма ответа, коды доступа.
- **Тесты доступа и безопасности.** Роли, аутентификация (сессия и токен), ограничение
  частоты, security-заголовки, аудит действий.
- **Нефункциональные.** Нагрузочное тестирование (Locust) и порог покрытия — вынесены в
  отдельные процедуры (разделы 6–7).

## 4. Область тестирования

**В объёме:** прикладная логика расчётов, контракты данных, API и его контракт, веб-страницы
и кабинет, локализация, разграничение доступа, воспроизводимость ML, эксплуатационные
эндпойнты, нефункциональные показатели (латентность, покрытие).

**Вне объёма:** корректность исходных данных Росстата (принимаются как есть), инфраструктура
облачного провайдера, нагрузочная устойчивость сторонних сервисов (PostgreSQL, Redis),
браузерный UI-автотест (проверяется вручную, см. security-review).

## 6. Тестовое окружение и инструменты

- **Прогон:** `pytest` + `pytest-django`; временная витрина DuckDB через фикстуры; SQLite
  для слоя OLTP в тестах.
- **Статический анализ:** `ruff` (линт + формат), `mypy` (строгая типизация прикладного кода).
- **Покрытие:** `pytest-cov`; отчёт по файлам с указанием непокрытых строк.
- **Нагрузка:** `locust` (сценарий `tests/load/locustfile.py`).
- **Непрерывная интеграция:** GitHub Actions — на каждый push/PR прогоняются линт, типизация,
  валидация схемы OpenAPI, полный набор тестов с порогом покрытия.

## 7. Критерии

**Критерии входа:** ветка собирается, зависимости установлены, витрина DuckDB доступна.

**Критерии выхода / приёмки (гейты CI):**

| Критерий | Порог | Факт |

| Все тесты проходят | 100% | 527/527 ✓ |

## 8. Управление дефектами

Дефекты фиксируются как GitHub Issues с уровнем серьёзности и приоритетом.

| Уровень | Значение |
|---|---|
| S1 — критический | Система или ключевая функция неработоспособна, обхода нет |
| S2 — высокий | Важная функция нарушена, обход затруднён |
| S3 — средний | Функция работает частично, есть обходной путь |
| S4 — низкий | Незначительный дефект, слабое влияние на пользователя |
| S5 — тривиальный | Косметика, опечатки, без влияния на функциональность |

Приоритет (P1–P3) задаёт срочность исправления и может не совпадать с серьёзностью.
Жизненный цикл: `Открыт → В работе → На проверке → Закрыт` (либо `Отклонён`/`Дубликат`).

## 9. Риски и меры

| Риск | Мера |
|---|---|
| Недетерминизм ML-моделей | Фиксация `seed`, разделение обучения (`write=True`) и сухих прогонов |
| Побочные записи в артефакты при тестах | Все тесты используют временные каталоги/БД; изоляция проверена |
| Регресс производительности | Нагрузочный замер и порог покрытия как гейты |
| Расхождение локализации | Тесты `test_i18n` фиксируют переводы ключевых строк |
| Скрытая деградация доступа | Тесты ролей и аутентификации фиксируют контракт публичных и приватных поверхностей |
