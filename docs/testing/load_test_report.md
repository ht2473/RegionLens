# Отчёт по нагрузочному тестированию

## 1. Цель

Подтвердить целевой показатель отклика **< 200 мс** для «горячих» read-only эндпойнтов API
(карта, дашборд региона, рейтинг, типология, переходы) при реалистичной конкурентной
нагрузке. Проверяется установившаяся производительность прикладного слоя поверх
предрасчитанной витрины DuckDB.

## 2. Методика

- **Инструмент:** Locust 2.44 (сценарий — `tests/load/locustfile.py`).
- **Сценарий:** имитация аналитика; веса задач отражают частоту реальных обращений —
  слой карты (5), дашборд региона (4), рейтинг индекса (3), типология (2), переходы (2),
  справочник регионов (1). Коды ОКАТО берутся из `/api/regions/` при старте, годы — из окна
  анализа 2010–2024, поэтому все запросы валидны (HTTP 200).
- **Сервер:** Gunicorn, 4 синхронных воркера, WSGI-приложение поверх собранной витрины
  DuckDB (~40 МБ), кэш ответов в памяти процесса (LocMemCache).
- **Профиль нагрузки:** 30 одновременных пользователей, разгон 30/с, длительность 30 с
  (≈ 50 запросов/с).
- **Прогрев:** перед замером выполнён прогрев (инициализация соединений DuckDB во всех
  воркерах и наполнение кэшей). Обоснование — в разделе 4.
- **Ограничение частоты** на время замера латентности ослаблено переменными окружения
  (`THROTTLE_ANON`, `THROTTLE_USER` — большое значение). Это необходимо потому, что все
  виртуальные пользователи Locust обращаются с одного IP и делят **единый анонимный лимит**
  (в проде — 120/мин на IP). При ~50 запр./с этот лимит исчерпывается за секунды, и дальнейшие
  запросы отклоняются с `429` — что измеряло бы работу ограничителя, а не латентность
  эндпойнтов. Поведение под действующими лимитами вынесено в отдельный раздел 5.

## 3. Результаты (установившийся режим)

Время отклика, мс; **0 отказов** на 1445 запросов; пропускная способность **49.8 запр./с**.

|--------------------------|---------:|--------:|----:|----:|----:|-----:|

Целевой порог **< 200 мс** выдержан по всем эндпойнтам вплоть до 99-го перцентиля
(агрегированные p95 = 69 мс, p99 = 160 мс). Единичные выбросы у верхней границы (макс 330 мс)
— остаточные «холодные» запросы к неохваченным кэшем комбинациям параметров.

## 4. О «холодном старте»

Первый прогон **без прогрева** показал в хвосте разовые всплески до ~7 с при неизменно
низкой медиане (8–18 мс). Причина — одновременный разгон пользователей на «холодном» сервере:
воркеры инициализируют read-only соединение с DuckDB и наполняют кэши ответов «на лету»,
и первые запросы каждого воркера оплачивают эту инициализацию. Это разовый эффект запуска,
не относящийся к установившейся нагрузке, поэтому замер в разделе 3 выполнен после прогрева.
В проде тот же эффект устраняется прогревом при деплое и общим кэшем (Redis).

## 5. Поведение ограничения частоты под нагрузкой

Отдельный прогон с **действующими прод-лимитами** (120/мин на IP, без ослабления) при том же
профиле (30 пользователей с одного узла, ~50 запр./с, 30 с) подтверждает работу ограничителя.

- Всего запросов: 1215; отклонено с `429`: 1095 (**90 %**).
- Единый анонимный лимит (120/мин на IP) исчерпывается за первые секунды, после чего избыточный
  поток с этого источника отклоняется — ограничитель ведёт себя как задумано.
- Успешно обслуженные запросы остаются быстрыми: медиана 8 мс, p95 ≈ 84 мс.

Это ожидаемый и желательный результат, а не дефект: все виртуальные пользователи идут с одного
IP и делят один лимит, тогда как реальные пользователи приходят с разных адресов, и каждому
доступны свои 120/мин (по токену — 600/мин). Таким образом, замер подтверждает две вещи
одновременно: прикладной слой быстр (раздел 3), а защита от злоупотреблений частотой —
действенна (этот раздел).

## 6. Вывод

Прикладной слой уверенно укладывается в целевые < 200 мс на установившейся нагрузке:
медиана ≈ 10 мс, p95 = 69 мс, p99 = 160 мс, отказов нет. Архитектура «предрасчёт → чтение из
DuckDB» обеспечивает стабильно низкую латентность горячих эндпойнтов.

## 7. Воспроизведение

```bash
# 1) Поднять сервер на :8000 с собранной витриной DuckDB.
#    Для замера латентности (раздел 3) ослабить лимит частоты:
THROTTLE_ANON=1000000/min THROTTLE_USER=1000000/min \
  python backend/manage.py runserver        # или gunicorn (см. §2)

# 2) Запустить замер (безголовый режим, 30 пользователей, 30 с)
locust -f tests/load/locustfile.py --host http://127.0.0.1:8000 \
       --headless -u 30 -r 30 -t 30s --csv results
# либо: make load  (интерактивный веб-интерфейс Locust)

# Для раздела 5 (поведение ограничителя) сервер запускается БЕЗ ослабления лимитов —
# тогда избыточный поток с одного IP ожидаемо отклоняется с 429.
```

> Замер выполнен на локальном окружении; абсолютные значения зависят от оборудования.
> Значимы порядок величин и запас относительно целевого порога, а не конкретные миллисекунды.
