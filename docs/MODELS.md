# Модели и обучающие данные

RegionLens использует две управляемые ML-модели. Они обучаются офлайн-конвейером
(`pipeline`), сохраняются в каталог `models/` вместе с карточкой метаданных и затем
загружаются приложением и применяются к профилю региона (страница «Модели» показывает
живую демонстрацию). Этот документ фиксирует формальные обозначения моделей и связь
каждой модели с её обучающими данными.

## Обозначения моделей

Каждая модель имеет говорящее имя (его используют код и витрина) и формальное
обозначение `model1` / `model2`. Обозначение — это алиас: при сохранении рядом с
`<говорящее_имя>.joblib` пишется байтовая копия `<обозначение>.joblib`, а само
соответствие хранится в карточке модели (поле `alias`) и в таблице ниже.

| Обозначение | Файлы                                                   | Алгоритм                        | Назначение                                             | Метрика качества                    |
| ----------- | ------------------------------------------------------- | ------------------------------- | ------------------------------------------------------ | ----------------------------------- |
| `model1`    | `model1.joblib` ≡ `typology_classifier.joblib` (+ `.json`) | `HistGradientBoostingClassifier` | Предсказание типа региона по его показателям            | Точность на кросс-валидации (`cv_accuracy`) |
| `model2`    | `model2.joblib` ≡ `anomaly_detector.joblib` (+ `.json`)    | `IsolationForest`               | Оценка нетипичности профиля региона на фоне всей панели | Доля выявленных аномалий (`anomaly_share`) |

`model1.joblib` байт-в-байт совпадает с `typology_classifier.joblib`, `model2.joblib` —
с `anomaly_detector.joblib`. Карточка (`.json`) пишется только под говорящим именем,
чтобы витрина моделей не показывала одну модель дважды; актуальные метаданные (алгоритм,
параметры, метрики, признаки, дата обучения) всегда в `typology_classifier.json` и
`anomaly_detector.json`.

## Обучающие данные

Обе модели обучаются на одной матрице признаков, которую готовит офлайн-конвейер.

- **Источник.** Показатели Росстата (муниципальная и региональная статистика),
  импортируемые в аналитическое хранилище DuckDB (`data/regionlens.duckdb`, версионируется
  через Git LFS). ETL-стадия конвейера строит справочники и факты (`metric_dim`,
  `region_dim`, `fact_region`).
- **Матрица признаков.** Стадия `features` формирует широкую матрицу `features_wide`:
  «ядровые» метрики, импутация пропусков и z-нормировка по годам. Признаки упорядочены по
  `metric_id` для детерминизма. Конкретный перечень признаков (идентификаторы метрик
  Росстата) сохранён в карточке каждой модели в поле `feature_names`.
- **Целевая переменная.**
  - `model1` (типология): метки — стабильные во времени `cluster_id` из кластеризации
    KMeans (стадия `typology`). Классификатор учится воспроизводить типологию по признакам,
    чтобы тип можно было предсказать для произвольного профиля без повторной кластеризации.
  - `model2` (аномалии): обучение без учителя — `IsolationForest` оценивает нетипичность
    профиля; целевых меток нет, доля аномалий задаётся параметром `contamination = 0.05`.
- **Объём выборки.** `n_samples = 1275` — 85 субъектов РФ × 15 лет (2010–2024). Точное
  значение по каждой модели указано в её карточке (`n_samples`).
- **Детерминизм.** Случайность зафиксирована: `seed = 42` (обе модели), `contamination = 0.05`
  (для `model2`). Значения — в карточке (`params`).

Таким образом, обучающие данные полностью сохранены и воспроизводимы: витрина DuckDB —
в репозитории (Git LFS), перечень признаков и объём выборки — в карточках моделей, а сами
модели пересобираются из этих данных одной командой.

## Воспроизводимость

Переобучение и запись всех артефактов (обе модели, карточки, копии-обозначения):

```bash
python -m pipeline.run_all            # весь конвейер (сырьё → витрина → модели)
python -m pipeline.run_all --from features   # пересобрать с матрицы признаков и до конца
```

(В Makefile — цель `make pipeline`.) Стадии `features` → `typology` готовят данные и метки,
после чего обучаются и сохраняются модели. Сохранение идёт через `pipeline.models_io.save_model`,
которое пишет `<имя>.joblib`, карточку `<имя>.json`, копию-обозначение `<alias>.joblib` и
best-effort регистрирует запуск в MLflow.

## Где что лежит

- `models/typology_classifier.joblib`, `models/typology_classifier.json` — модель `model1` и её карточка.
- `models/anomaly_detector.joblib`, `models/anomaly_detector.json` — модель `model2` и её карточка.
- `models/model1.joblib`, `models/model2.joblib` — копии под формальными обозначениями.
- `data/regionlens.duckdb` — аналитическая витрина с обучающими данными (Git LFS).
- Соответствие обозначений задаётся в `pipeline/models_io.py` (`MODEL_ALIASES`).
